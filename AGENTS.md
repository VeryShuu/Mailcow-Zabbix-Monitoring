# Repository Guidelines

## Project Overview

Mailcow Monitoring v1.0 — a complete Zabbix monitoring solution for [Mailcow-Dockerized](https://mailcow.email/). Provides 307 UserParameters, 29 collector modules, 71 triggers, and 19 dashboards. Licensed under GPLv3.

## Architecture

```
systemd timer (60s) → mailcow-collector.py (root)
  → /var/tmp/mailcow-monitor.json          (main cache, 60s)
  → /var/tmp/mailcow-monitor-slow.json     (slow cache, 1h)
  → /var/tmp/mailcow-monitor-mailflow.json (mailflow cache, 5m)
  → /var/tmp/mailcow-monitor-version.json  (version cache, 1h)
    → Zabbix Agent 2 (zabbix user) → mailcow-reader.sh → reads JSON
```

- **`scripts/mailcow-collector.py`** — main Python collector, runs as root, implements 29 monitoring modules. Single entry point for all metric collection.
- **`scripts/mailcow-reader.sh`** — JSON reader called by every UserParameter; uses embedded Python3 for safe value extraction (no shell injection).
- **`scripts/mailcow-common.sh`** — shared shell library sourced by `check_*.sh` scripts; provides MySQL helper `_mysql()` and `get_mail_domains()`.
- **`mailcow-zabbix.conf`** — all 307 Zabbix `UserParameter` definitions in one file; installed to `/etc/zabbix/zabbix_agent2.d/mailcow.conf`.
- **`templates/mailcow-complete-monitoring.yaml`** — Zabbix 7.0 template (256 KB); import via Zabbix UI.
- **`scripts/check_*.sh`** — helper shell scripts for slow-cache checks (DNS, TLS, RBL, PTR, open relay, security audit).
- **`scripts/dovecot_check.sh`** — Dovecot stats via container logs (cached 60s).
- **`scripts/postfix_log_analysis.sh`** — Postfix log + Postscreen event counting (cached 60s).
- **`scripts/postfix_stats_docker.sh`** — Postfix queue stats from container.
- **`scripts/sync_jobs_check.sh`** — IMAP sync job status via MySQL.
- **`mailcow-monitor.service` / `.timer`** — systemd oneshot service triggered every 60 seconds.
- **`/etc/mailcow-monitor.conf`** — runtime config generated by `install.sh`; sets `MAILCOW_DIR`, `BACKUP_PATH`, `SKIP_GIT_FETCH`.

The Zabbix Agent never touches Docker or MySQL directly — it only reads the pre-generated JSON file. No `UnsafeUserParameters`, no sudo.

## Build, Test, and Development Commands

**Installation (requires root, Debian/Ubuntu host):**
```bash
sudo ./install.sh
```

**Validation — test all 307 UserParameters via `zabbix_get`:**
```bash
sudo ./test-complete.sh
```

**Uninstall:**
```bash
sudo ./uninstall.sh
```

**Force immediate metric collection:**
```bash
systemctl start mailcow-monitor.service
systemctl restart zabbix-agent2
```

**Run collector manually (debug):**
```bash
sudo python3 /usr/local/bin/mailcow-collector.py
```

**Spot-check a single key:**
```bash
zabbix_get -s 127.0.0.1 -k mailcow.rspamd.scanned
```

**Inspect collected JSON:**
```bash
cat /var/tmp/mailcow-monitor.json | python3 -m json.tool | head -20
```

**Check agent logs:**
```bash
tail -20 /var/log/zabbix/zabbix_agent2.log
```

## Coding Style & Naming Conventions

- **Python** (`mailcow-collector.py`): standard Python 3, `shell=False` for `run_cmd()` where possible (security + performance). Passwords passed via env vars (`MYSQL_PWD`), never on the command line.
- **Shell scripts**: `#!/bin/bash`, use `set -e` in installer. Color constants (`RED`, `GREEN`, etc.) at top of interactive scripts.
- **Zabbix key naming**: `mailcow.<module>.<metric>` (e.g., `mailcow.rspamd.scanned`, `mailcow.disk.root.used`). Postfix keys use `postfix.<metric>` prefix. Agent keys use `zabbix.agent.<metric>` prefix.
- **No `UnsafeUserParameters`** — all UserParameters must be safe and not require shell injection.
- **Collector JSON keys** must match the keys referenced in `mailcow-zabbix.conf` exactly.
- **No comments in code** unless explicitly requested.

## Key Implementation Patterns

- **Slow cache** (`/var/tmp/mailcow-monitor-slow.json`): expensive checks (TLS, DNS, RBL, security audit) cached separately; parallelized with `ThreadPoolExecutor` (6 threads).
- **Mailflow cache** (`/var/tmp/mailcow-monitor-mailflow.json`): 5-minute cache, separate from the 1-hour slow cache, for fresher data.
- **Version cache** (`/var/tmp/mailcow-monitor-version.json`): git fetch once per hour; skippable via `SKIP_GIT_FETCH=true` in config.
- **Container discovery**: single `find_all_containers()` call per collector run, not per module. Discovers: postfix, dovecot, rspamd, netfilter, clamd, watchdog, memcached, redis, mysql, nginx, sogo, unbound.
- **LLD rules**: 4 discovery rules — domains, mailboxes, sync jobs, Docker containers. Discovery keys follow `mailcow.lld.<type>` pattern.
- **Anomaly detection**: 5 `trendavg()`-based triggers in the Zabbix template (no hardcoded thresholds); do not replace with static values.
- **Atomic writes**: all JSON files written to `.tmp` first, then `os.rename()` for crash safety.
- **Error tracking**: each module is wrapped in try/except; errors reported via `mailcow.collector.errors` and `mailcow.collector.error.detail`.
- **MySQL dual-auth**: tries `DBROOT` first for SHOW GLOBAL STATUS, falls back to `DBPASS` (mailcow user).
- **ClamAV**: single `docker exec` with compound shell command instead of 3-5 individual calls.
- **Docker Health**: targeted `docker stats` + one batched `docker inspect` instead of N individual calls.

## Requirements

Target system: Linux (Debian/Ubuntu), Mailcow-Dockerized, Zabbix 7.0 + Zabbix Agent 2, Python 3, `pflogsumm`, `dig` (dnsutils), `openssl`, `netcat`.
