## ПРОИЗВОДИТЕЛЬНОСТЬ

### P1. Reader: заменить fork Python3 на jq / JSONPath

- **Статус**: ⬜ Не реализовано
- **Сложность**: Средняя (переписать reader + возможные изменения шаблона)
- **Описание**: Сейчас каждый цикл опроса Zabbix вызывает 307 UserParameter'ов, и каждый из них запускает цепочку: fork bash → fork python3 → парсинг всего JSON-файла (~300 КБ) → извлечение одного ключа → выход. Это ~600 процессов и ~307 полных разборов JSON **каждую минуту**. Это самое узкое место во всей системе. Исправление заменяет Python3-heredoc в `mailcow-reader.sh` на один вызов `jq` (быстрее в ~10 раз, нет времени запуска интерпретатора). Ещё лучший вариант — перевести шаблон Zabbix на один master-item `vfs.file.contents` с зависимыми элементами и JSONPath-предобработкой — это полностью убирает все 307 fork'ов.
- **Изменения в шаблоне Zabbix**: **ДА** — при подходе с master-item все 307 элементов нужно переделать в зависимые с JSONPath-предобработкой. При замене только на jq — шаблон не меняется.
- **Риски**:
  - Вариант с `jq`: минимальный риск, замена «один к одному». Требует `jq` как новую зависимость.
  - Вариант с master-item: полная переработка шаблона. История данных сохраняется (ключи не меняются), но нужен реимпорт шаблона. **Большой объём работы, потери данных нет.**
  - Если `jq` не установлен, а Python3-fallback убран — reader перестанет работать.
- **Снижение рисков**: Оставить Python3 как запасной вариант в reader. Проверять наличие `jq` перед использованием.

### P3. Python читает кэш-файлы shell-скриптов напрямую (Dovecot, Postfix logs, Sync)

- **Статус**: ⬜ Не реализовано
- **Сложность**: Лёгкая (30 строк Python на модуль)
- **Описание**: `collect_postfix_logs()` вызывает shell-скрипт `postfix_log_analysis.sh` 20 раз — по разу на каждую метрику (sasl_auth_failed, relay_denied, rbl_reject и т.д.). Каждый вызов запускает bash, читает кэш-файл и извлекает одно значение. Та же проблема с `collect_dovecot()` (9 вызовов `dovecot_check.sh`) и `collect_sync()` (6 вызовов `sync_jobs_check.sh`). Итого: 35 лишних fork'ов bash за один запуск коллектора. Исправление читает JSON-кэши (`/var/tmp/postfix_log_analysis.cache`, `/var/tmp/dovecot_check.cache`) напрямую из Python через `json.load()` — одно чтение файла вместо 20 запусков подпроцессов.
- **Изменения в шаблоне Zabbix**: **НЕТ**
- **Риски**: Низкие. Если формат кэш-файла изменится в shell-скрипте, а Python-код не обновят — метрики вернут значение по умолчанию (0). Сами shell-скрипты по-прежнему должны запускаться (для наполнения кэша).
- **Снижение рисков**: Оставить один вызов shell-скрипта для обновления кэша, затем читать результат напрямую.

## АРХИТЕКТУРА

### A1. Перенос сбора Dovecot/Postfix-log/Sync в Python-коллектор

- **Статус**: ⬜ Не реализовано
- **Сложность**: Средняя (переписать 3 shell-скрипта как Python-функции, ~200 строк)
- **Описание**: Коллектор уже имеет полный доступ к Docker и MySQL, но по-прежнему делегирует сбор статистики Dovecot, разбор логов Postfix и проверку sync-задач внешним shell-скриптам. Каждый вызов shell-скрипта запускает bash, заново ищет Docker-контейнер и выполняет свои запросы. Это избыточно, так как коллектор уже знает все имена контейнеров и имеет доступ к MySQL. Перенос в Python устраняет ~35 вызовов подпроцессов за один запуск, убирает 3 внешние зависимости и делает код проще для тестирования и отладки. Shell-скрипты останутся в репозитории для автономного использования, но коллектор их не будет вызывать.
- **Изменения в шаблоне Zabbix**: **НЕТ** (те же ключи и значения метрик)
- **Риски**: Средние. Shell-скрипты протестированы в продакшене. Переписывание создаёт потенциал для новых ошибок в регулярных выражениях парсинга логов, доступе к контейнерам или MySQL-запросах.
- **Снижение рисков**: Запустить обе реализации параллельно на 1 неделю и сравнить результаты. Оставить shell-скрипты как запасной вариант. Добавить юнит-тесты для новых Python-функций.
